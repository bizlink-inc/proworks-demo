## 1. 会議概要

## 2. UI仕様・デザイン関連

### 2.1 デザイン準拠方針
- **デザイン仕様書（UI使用書）に忠実に沿う方針で確定。**
  - 佐藤：「基本的には仕様書通りに再現中だが、抜けがある部分は補完している」
  - 鈴木：「Bizlink側の提案も歓迎だが、最終的にはUI使用書ベースで統一したい」

### 2.2 指摘・修正点
| 項目 | 内容 | 対応 |
|------|------|------|
| 表示位置のずれ | ボタン・文字位置が左寄り／中央寄せなど異なる箇所有り | 修正対応 |
| 「メニュー」表記 | 使用書には不要。削除対応 | 佐藤対応済 |
| ボタン角丸・枠線 | 丸角の有無など、統一感が不足 | 使用書準拠で統一 |
| 文字数制限 | 案件カードで2行崩れが発生 | 文字数制限または改行制御を追加 |
| アイコン差異 | Font Awesomeのアイコンが異なる・サイズ不一致 | 指定コードへ統一（有料版要確認） |
| 応募済み案件の表示 | 応募済み案件を一覧から除外する仕様 | 実装ロジックで制御（未反映→修正へ） |

### 2.3 実装方針
- 細部仕様（色・位置・角丸・文字制限）は**一覧で具体的に共有**してもらい順次反映。  
- 佐藤氏：「スピード重視でまず実装 → 鈴木氏レビュー後に微修正」で合意。

## 3. 応募済み案件の表示ロジック

- 応募済み案件は「案件一覧」には**表示しない仕様**。
- 現状、応募済み案件が重複表示されている（一覧＋応募済み画面）。
- 佐藤：「仕様通りにロジックを分離し修正対応予定。」

## 4. モバイル・レスポンシブ対応

### 4.1 対応方針
- Next.js標準のレスポンシブ対応を活用し、**幅縮小での自動調整を基本方針**。
- ただし、モバイル専用デザイン（UI使用書上部記載）に明示されている部分のみ個別対応。

### 4.2 合意点
| 項目 | 内容 |
|------|------|
| タブメニュー構成 | UI使用書のモバイルレイアウトに準拠（タブ＋ハンバーガー） |
| 応募済み案件ページ | 2行表示対応を許容（幅制限で折返し） |
| 未定義画面 | ページ不足（退会・お問い合わせ）は新規作成必要 |
| 要望 | 各ページのモバイル版デザインもUI使用書に追加してほしい（佐藤要望） |

## 5. 退会フロー設計

- 基本設計書「通知メール文章」シート10〜11行目を参照。
- **退会フロー概要：**
  1. 「退会申請」ボタン押下  
  2. ポップアップ表示 → チェック3項目（確認事項）  
  3. 「退会手続きへ進む」ボタン  
  4. 再度ポップアップ：「本当に退会しますか？」→ 実行

→ **チェックボックス付きの2段階確認構成**で合意。

---

## 6. 通知機能（お知らせ・メンテナンス）

### 6.1 表示仕様
- 画面上部に**赤帯の固定通知バー**を表示。
- 表示内容：メンテナンス・障害・お知らせ。

### 6.2 実装方式
- **kintoneで通知データを管理。**
  - 種別：プルダウン選択（メンテナンス／障害／お知らせ）
  - 本文：リッチテキスト入力
  - 掲載期間：カレンダーで日付選択
- 表示中はフロント側が自動反映。

→ シンプルなkintone構成（3フィールド）で管理する方針。

---

## 7. インフラ構成・メールサーバー

### 7.1 環境方針
- **GCP（Google Cloud Platform）で構築確定。**
- Gemini APIを利用したAI処理を同環境で稼働。

### 7.2 メールサーバー問題
| 項目 | 内容 |
|------|------|
| GCP内メールサービス | GCPには標準のメールサーバー機能なし |
| 対応案① | 外部サービス **SendGrid**（推奨）を利用<br>→ 1日100通まで無料、月12,000通無料枠 |
| 対応案② | 自前構築（Postfix等）<br>→ 鈴木氏「Docker／Ubuntu構築なら可能だが工数増」 |
| 結論 | 当面はSendGridで対応。将来的に自前構築も検討余地あり。 |

## 9. 決定事項まとめ

- UIは**デザイン仕様書に完全準拠**。  
- 応募済み案件は一覧非表示で制御。  
- モバイル対応は**基本レスポンシブ＋一部専用UI**。  
- 通知はkintone管理、上部赤帯で表示。  
- メール送信は**SendGrid採用**、将来自前サーバー化も可。
