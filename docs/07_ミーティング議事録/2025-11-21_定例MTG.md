## 1. 会議概要

今回の定例では、ProWorksにおける**AIマッチング機能の設計指針**と、  
**スキルマッチングのロジック構築**に関する詳細な技術検討を実施。

特に、  
- コスト最適化（AI利用頻度・処理単位）  
- kintone内のデータ構造と負荷  
- 案件・人材スキル情報の正規化方法  
- 今後のAI連携（CSV or API）構想  
が主要論点。

---

## 2. AIマッチングの基本設計方針

### 2.1 現在の想定フロー（鈴木 → 佐藤）
1. 案件情報（作業工程・環境・必須スキルなど）をテキスト入力。  
2. 対象人材のPDFをAIに渡して分析。  
3. AIがスコアリング（例：80点以上を提案対象）を出力。  
4. 提案対象者をProWorks上に表示。

→ 実質的には「案件 × 各人材」の1対1比較を行うプロンプト構成。

### 2.2 処理構造の懸念点（佐藤）
- 登録人材が1,000名規模の場合、**1案件につき1,000回AI呼び出し**が必要。  
- 精度を維持するには「1対1スコアリング」が最適だが、**コスト負担が膨大**。  
- 複数人を1プロンプトにまとめる方法も可能だが、**精度が著しく低下**する。

### 2.3 コスト方針（末吉）
- 「AIは常時自動で回すのではなく、**必要時のみ手動実行**」とする。  
- 日常的なマッチングはプログラムロジックで完結。  
- 緊急性の高い案件・重要提案時のみAIマッチを実行。  
- マッチング対象も**プログラムで抽出した上位候補（例：10名程度）に限定**。

→ 結論：  
　AIマッチは “限定的・選択的運用” とし、**AIコストの最小化**を徹底。

---

## 3. プログラムマッチ（機械的ロジック）設計

### 3.1 抽出対象
- kintoneに登録された人材情報を対象に検索。
- 案件の「職種・スキルタグ」を基準に部分一致検索。
- 例：案件が「AWS, JavaScript, React」の場合、同タグを持つ人材を抽出。

### 3.2 ロジック検討
- 案件ごとにフラグ付与する実装は負荷が大きいため避ける方向。  
- 代替案：
  - 案件×人材のマッチ情報を保持する**専用DB（中間テーブル）**を新設。
  - 案件ID・人材IDを紐付け、マッチ候補10名を格納。
  - 各候補者に対してAIスコア付与可。

### 3.3 今後の課題
- 「どの基準で上位候補とするか」（単語一致数 or 重みづけ）の仕様を決定。  
- ストームストーリー社の過去ロジックを参照し、最適な判定式を再定義する。

## 4. スキルマッチング設計（タグ構造）

### 4.1 現状課題
- 人材側スキルはテキスト記述が多く、単語抽出が困難。  
- 案件側スキルはフィールド構造が限定的でマッチ精度が低い。

### 4.2 改善方針
1. **案件側の「職種ポジション」「案件特徴」項目を拡張**
   - 例：「AWS」「GCP」「Ruby on Rails」「Next.js」などを選択肢化。
   - 選択肢を固定することで正規化と自動抽出が容易に。
2. **人材側スキルはテキストから自動抽出**
   - 登録時にAIが経歴文からキーワードを抽出。
   - 案件側タグと突き合わせて一致数をスコア化。

### 4.3 実装提案
- 案件登録時、**選択式で技術タグをチェック入力**（人手補助）。  
- 将来的には、**AIプロンプトで案件文→CSV自動生成→kintoneインポート**を目指す。

---

## 5. CSV・API連携構想

### 5.1 CSVインポート構想（鈴木案）
- 案件概要をプロンプトに入力 → AIがCSV出力。  
- 出力CSVをそのままkintoneにインポート。  
- 項目（職種／ポジション／スキル／担当者など）を自動整形。  
- 入力作業の効率化を狙う。

### 5.2 API自動登録構想（佐藤案）
- CSV経由せず、**ChatGPT/Gemini → kintone API 直書き**の形が理想。  
- 実現できれば、完全自動化が可能。  
- まずはプロンプト精度を検証 → 問題なければAPI化。

→ **最終目標：案件情報を投げるだけで、自動的にkintone登録完了。**

## 6. インフラ関連補足（WordPress連携）

- アルマ社内でのWordPress／kintone連携構成についてヒアリング実施。  
- 現状：
  - メディアはWordPress（別サーバー／A2 Hosting等）で運用。  
  - kintoneは独立DBとして稼働。  
  - 画像はおそらくS3等バケット保管（確認中）。  
- 鈴木氏より懸念：
  - 異サーバー間通信時の**ログイン遅延／APIレイテンシー**。
  - Google Cloud Run等、**GCP内完結構成への統合案**を検討。

- **AIマッチングの自動化範囲は限定**。  
- **コスト抑制重視／候補者絞り込み前提**でAIを利用。

- CSV→API自動登録構想で、運用負荷の大幅削減を目指す。  
- 次回はスキルタグ定義とAIマッチロジックの初版レビューを実施予定。
